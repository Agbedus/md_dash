Build a simplified attendance module for an existing FastAPI application.

Do not rebuild authentication, user accounts, or session management.
Those already exist.

Use existing roles:
- staff
- manager
- super_admin

Only build the attendance and office presence logic.

==================================================
1. CORE PURPOSE
==================================================

The system must do the following:

- detect whether a staff user is in the office based on location
- automatically clock staff in when they are confirmed in the office during the morning check-in window
- automatically clock staff out at the end of the workday
- keep attendance separate from login/session state
- allow managers and super_admins to manually override attendance records
- keep an audit trail of overrides

Do not treat user login as clock-in.
A user can stay logged in while being clocked out.

==================================================
2. SIMPLE BUSINESS RULES
==================================================

Use one office location and one active attendance policy for the MVP.

Office distance rules:
- 0 to 5 meters from office = IN_OFFICE
- more than 5 to 15 meters = TEMPORARILY_OUT
- more than 15 meters = OUT_OF_OFFICE

Work time rules:
- work start time = 08:30
- work end time = 18:00
- check-in window = 07:30 to 10:00
- auto clock-out time = 18:00

State confirmation rules:
- do not change attendance state from a single GPS ping
- confirm TEMPORARILY_OUT only after 5 minutes of consistent evidence
- confirm OUT_OF_OFFICE only after 10 minutes of consistent evidence
- confirm return to IN_OFFICE only after 2 minutes of stable evidence

GPS quality rule:
- if accuracy_meters > 30, save the location log
- but do not use that single sample alone to move the user into OUT_OF_OFFICE

==================================================
3. ATTENDANCE RULES
==================================================

Presence state values:
- IN_OFFICE
- TEMPORARILY_OUT
- OUT_OF_OFFICE

Attendance state values:
- NOT_CLOCKED_IN
- CLOCKED_IN
- CLOCKED_OUT

Auto clock-in rule:
Clock a user in when all are true:
- user is staff
- user is not already clocked in for today
- current time is between 07:30 and 10:00
- confirmed presence state is IN_OFFICE

When auto clock-in happens:
- create today’s attendance record if it does not exist
- set attendance_state = CLOCKED_IN
- set clock_in_at = current timestamp
- set first_seen_in_office_at if it is empty

Auto clock-out rule:
At 18:00, if a user is still CLOCKED_IN:
- set attendance_state = CLOCKED_OUT
- set clock_out_at = 18:00 for that work date

Do not reopen attendance automatically after a user has been CLOCKED_OUT for the day.

Location update rule:
When confirmed presence is IN_OFFICE:
- update last_seen_in_office_at on today’s attendance record

==================================================
4. PERMISSIONS
==================================================

staff:
- send location updates
- view own attendance for today
- view own attendance history

manager:
- all staff permissions
- view team attendance
- override attendance records

super_admin:
- all permissions
- manage office location
- manage attendance policy
- override attendance records

Only manager and super_admin can perform overrides.

7. REQUIRED BACKEND LOGIC
==================================================

For every location update:

Step 1:
Get the authenticated user from the existing auth system.

Step 2:
Load the active office location and attendance policy.

Step 3:
Calculate distance from office on the server.
Do not trust the client for distance or zone decisions.

Step 4:
Derive the raw zone from the current point:
- distance <= 5 => IN_OFFICE
- distance > 5 and <= 15 => TEMPORARILY_OUT
- distance > 15 => OUT_OF_OFFICE

Step 5:
Save the location in location_logs.

Step 6:
Check recent location behavior and current active presence state.

Step 7:
Apply grace-period confirmation rules:
- only confirm TEMPORARILY_OUT after 5 minutes
- only confirm OUT_OF_OFFICE after 10 minutes
- only confirm IN_OFFICE return after 2 minutes

Step 8:
If confirmed state changes, write to presence_state_history.

Step 9:
Apply attendance rules:
- if confirmed IN_OFFICE during check-in window and user is not clocked in, auto clock in
- if confirmed IN_OFFICE, update last_seen_in_office_at
- if time is at or after 18:00 and today’s record is still CLOCKED_IN, auto clock out

==================================================
8. IMPLEMENTATION RULES
==================================================

Use thin route handlers.
Put business logic into services.

Create these services:
- location_service
- presence_service
- attendance_service
- override_service

location_service responsibilities:
- calculate haversine distance
- derive raw zone
- save raw location log

presence_service responsibilities:
- determine confirmed presence state
- apply grace periods
- manage state transitions
- write presence_state_history

attendance_service responsibilities:
- create today’s attendance record
- auto clock-in
- update first_seen_in_office_at
- update last_seen_in_office_at
- auto clock-out

override_service responsibilities:
- permission checks
- apply override changes
- save audit record in attendance_overrides

==================================================
9. IMPORTANT CONSTRAINTS
==================================================

- do not modify authentication code
- do not create a new users table
- do not link attendance directly to session state
- do not clock users in from login alone
- do not clock users out from one noisy GPS sample
- do not allow staff users to override attendance
- do not allow attendance to reopen automatically after clock_out_at has been set for that day
- do not skip audit logging for overrides

==================================================
10. ACCEPTANCE CRITERIA
==================================================

The module is complete when all of the following are true:

- attendance-related tables are created
- one active office location can be stored
- one attendance policy can be stored and edited
- location updates are accepted from authenticated staff users
- server calculates distance from office
- raw location logs are stored
- confirmed presence state uses grace periods
- users are auto clocked in when confirmed IN_OFFICE during the check-in window
- users are auto clocked out at 18:00
- only one attendance record exists per user per day
- staff can view their own attendance
- managers can view team attendance
- managers and super_admins can override attendance
- all overrides are audited
